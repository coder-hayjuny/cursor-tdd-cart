# Cart Discount TDD Practice

## 목적

이 프로젝트는 **장바구니 할인 계산 로직**을 TDD(Test-Driven Development) 방식으로 구현하는 연습 프로젝트입니다.

도메인 로직은 `src/cart.py`의 **Entity** 계층에 둡니다. 테스트와 구현은 **계약 ID(`INV-*`, `E-*`)** 를 기준으로 작성·추적합니다. 계약 ID는 테스트와 구현을 잇는 **추적의 못**입니다.

상세 요구사항·MomTest 근거는 [docs/PRD.md](docs/PRD.md)를 참고하세요.

**현재 단계:** RED 준비 — 계약별 실패 테스트 작성 전·작성 중. 구현은 아직 완료되지 않았습니다.

## 핵심 원칙

1. **ID에 없는 동작은 만들지 않는다.** 계약 표에 없는 할인·검증·기능은 구현하지 않습니다.
2. **테스트가 먼저다.** 구현보다 실패하는 테스트가 항상 앞섭니다.
3. **RED → GREEN → REFACTOR** 순서를 따릅니다.
   - RED: `tests/`만 수정, `src/`는 건드리지 않습니다.
   - GREEN: 해당 계약 ID를 만족하는 최소 구현만 추가합니다.
   - REFACTOR: 전부 통과한 뒤에만 구조를 정리합니다.
4. **과잉 구현을 금지한다.** 요청·계약에 없는 기능을 미리 만들지 않습니다.

## 계약 ID 목록

| ID    | 계약(불변식 / 에러)                                                 | 근거 레벨 | 계층        |
| ----- | ------------------------------------------------------------ | ----- | --------- |
| INV-1 | `subtotal(items) == Σ(price × qty)`                          | —     | Entity    |
| INV-2 | `amount ≥ 50000 → round(amount×0.9)` / `< 50000 → 그대로` 경계 포함 | L1    | Entity    |
| INV-3 | `final = 문턱할인 적용 후, VIP면 round(×0.95)`. 순서 문턱→VIP 고정         | L2    | Entity    |
| INV-4 | 모든 입력에서 `0 ≤ final_total ≤ subtotal`. 할인은 금액을 늘리지 않는다        | L3    | Entity    |
| E-1   | `items is None → TypeError`                                  | L0    | Boundary* |
| E-2   | `price` 또는 `qty`가 음수 → `ValueError`, 인덱스 포함                  | L0    | Boundary* |

## 계약 ID 설명

### INV-1

품목 목록 `items`의 소계는 각 품목의 `price × qty` 합과 같아야 합니다. 할인 적용 전 기준 금액입니다.

### INV-2

소계 `amount`가 **50,000원 이상**이면 `round(amount × 0.9)`를 적용하고, **50,000원 미만**이면 할인 없이 그대로 둡니다. 경계값 `50,000`은 할인 대상에 **포함**합니다.

### INV-3

최종 금액 `final`은 **문턱 할인(INV-2)을 먼저** 적용한 뒤, VIP 고객이면 그 결과에 `round(× 0.95)`를 적용합니다. 적용 순서는 **문턱 → VIP**로 고정합니다.

### INV-4

어떤 유효한 입력에서도 `final_total`은 0 이상이며 `subtotal` 이하여야 합니다. 할인 로직은 결제 금액을 키우지 않습니다.

### E-1

`items`가 `None`이면 `TypeError`를 발생시킵니다. 장바구니 데이터가 없는 호출을 조용히 처리하지 않습니다.

### E-2

어떤 품목의 `price` 또는 `qty`가 음수이면, 해당 **인덱스**를 포함한 `ValueError`를 발생시킵니다.

## 계층 의미

### Entity

`src/cart.py`가 담당합니다. Flask 등 외부 프레임워크를 import하지 않는 **순수 도메인 계산·검증** 로직입니다. `INV-1` ~ `INV-4`는 이 계층의 불변식입니다.

### Boundary*

`E-1`, `E-2`는 원래 **입력 검증 경계**(폼·API)에 가까운 규칙입니다. 현재 실습에서는 별도 Boundary 모듈 없이 **도메인 함수 진입점**(`cart.py`)에서 검증합니다. 테스트는 `tests/test_cart.py`에서 계약 ID별로 작성합니다.

## 예상 파일 구조

```text
.
├── README.md
├── docs/
│   └── PRD.md
├── src/
│   └── cart.py
└── tests/
    └── test_cart.py
```

## TDD 진행 순서

1. **RED** — 계약 ID별로 **실패하는** 테스트를 `tests/test_cart.py`에 작성합니다. 이 단계에서는 `src/cart.py`를 수정하지 않습니다.
2. **GREEN** — 해당 ID를 만족하는 **최소** 구현만 `src/cart.py`에 추가합니다. 구현 줄에는 충족한 계약 ID를 주석으로 남깁니다. (예: `# INV-2`)
3. **REFACTOR** — 모든 테스트가 통과한 상태에서만 구조를 개선합니다. 리팩터 전후 `pytest -q`로 동작 불변을 확인합니다.

권장 RED 순서: `E-1`, `E-2` → `INV-1` → `INV-2` → `INV-3` → `INV-4`

## 테스트 실행

```bash
pytest -q
```

`-q`는 **quiet mode**로, 통과·실패 요약만 간단히 보여 줍니다.

## 구현 금지 사항

- **할인 정책 추가** (계약에 없는 문턱·비율·조건)
- **쿠폰, 세금, 배송비, 포인트** 기능
- **계약 ID에 없는** 예외 처리·기본값 보정
- **UI, CLI, DB, API** 코드

*계약 ID에 없는 동작은 구현하지 않습니다.*
